66 single-warm-iter captures (22 queries x 3 GPU configs). 8 OOM at SF=100 (enc q9; dec q1,5,6,8,9,18,21). The OOM .nsys-rep files exist (~430KB each, nsys session metadata only) but contain ZERO CUPTI events because the std::bad_alloc fires at the allocator level (e.g. cudf::concatenate requesting 36GB on a 40GB A100) BEFORE any kernel launches. Full-process recapture also produces 0 events for the same reason. Aggregate over the 15-query intersection (queries succeeding in all 3 configs): native kernel=4515ms / sync=18935ms / h2d=73GB; enc_region kernel=4731ms (0.95x) / sync=20931ms / h2d=423MB; dec_region kernel=3669ms (1.23x) / sync=18163ms / h2d=8GB. Note kernel-time speedups are MUCH weaker at SF=100 than SF=20 (where enc=1.45x and dec=1.49x). H2D bytes drop is identical to SF=20 (~170x enc, ~9x dec), so the wall-time win at SF=100 (1.42x enc / 1.41x dec) is dominated by the H2D-on-host-side savings, not by GPU kernel-time savings. Sync time still dominates kernel time everywhere (sync/k = 4-5x at SF=100, vs 3-9x at SF=20).
